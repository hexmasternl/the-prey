## Context

The Prey is a backend-only ASP.NET Core 10 modular monolith (no MediatR, Minimal APIs, CQRS via `ICommandHandler`/`IQueryHandler`). Games are persisted in PostgreSQL via EF Core. The existing games domain supports lobby management but joining currently has no code-validation gate. The mobile client (Ionic/Angular, built separately) will handle deep-link resolution and routing; this design covers only the backend contract.

## Goals / Non-Goals

**Goals:**

- Assign a cryptographically random 8-digit numeric join code to every game at creation time.
- Add a `POST /games/{gameId}/join` endpoint that validates the join code before adding the caller to the lobby.
- Expose the join code in the game retrieval DTO so the owner can share it.
- Document the deep-link URL scheme (`theprey://join?gameId=<id>`) as the standard sharing format.
- Add EF Core migration for the `JoinCode` column.

**Non-Goals:**

- Deep-link resolution on the server (that is purely client-side routing).
- Join-code rotation or expiry (out of scope; codes live as long as the game is in Lobby state).
- Rate-limiting brute-force attempts (can be addressed separately via infrastructure middleware).
- SMS/email delivery of the code (sharing is the owner's responsibility).

## Decisions

### 1 — 8-digit numeric code generated in the domain, not the database

**Decision:** The `JoinCode` is generated by the `CreateGameCommand` handler using `Random.Shared.Next(10_000_000, 99_999_999).ToString()` and stored as a string.

**Rationale:** Keeps generation logic testable and deterministic in unit tests (mockable `IRandomCodeGenerator` interface if needed). Using a plain string avoids leading-zero ambiguity compared to `int`. 8 digits give 90 million possible values — sufficient for the number of simultaneous live games anticipated.

**Alternative considered:** Database sequence / identity — rejected because it leaks an ordered pattern and is harder to test.

### 2 — Join code validated inside the `JoinGameCommand` handler

**Decision:** The handler fetches the game by `gameId`, asserts it is in Lobby state, then compares the supplied `joinCode` to `game.JoinCode` using a constant-time string compare. Mismatches return a domain error surfaced as HTTP 400.

**Rationale:** Keeps business rules in the domain layer, consistent with the existing `ICommandHandler` CQRS pattern. No separate "validate code" endpoint — one round trip.

### 3 — Join code included in the game DTO (owner and participants can see it)

**Decision:** `JoinCode` is added to `GameDto` so any authenticated retriever (owner or lobby member) receives it. The owner can copy-paste or re-share it.

**Alternative considered:** Return join code only to owner — rejected for simplicity; the code alone is not a secret (the game must still be in Lobby state and the requester must authenticate).

### 4 — Deep-link URL format is documented, not enforced server-side

**Decision:** The canonical shareable URL is `theprey://join?gameId=<id>`. The backend does not issue or redirect these URLs; the client constructs them. A future RESTful redirect endpoint (`GET /join/{gameId}`) can be added later as a web-fallback.

**Rationale:** The server's sole responsibility is validating the join code on `POST /games/{gameId}/join`; the link is a UX concern.

### 5 — New `JoinGameCommand` / `JoinGameCommandHandler` — existing `JoinLobbyCommand` is unchanged

**Decision:** Introduce a new command `JoinGameCommand { GameId, JoinCode, UserId }` handled by `JoinGameCommandHandler`. The existing lobby-join path (if any internal/owner bypass exists) is kept separate to avoid breaking owner flow.

**Rationale:** Single-responsibility; avoids patching an existing handler with conditional logic.

## Risks / Trade-offs

- **Code collision** — two concurrent games could get the same code. At the anticipated scale (hundreds of simultaneous games) the 90 M key space makes collision negligible; no uniqueness constraint is added to the column to avoid migration complexity.
- **Brute-force** — 8 digits are guessable if not rate-limited. Mitigation: apply API-level rate limiting per IP/user on the join endpoint (infrastructure concern, out of scope here but documented).
- **No code rotation** — if a code leaks before the game starts, the owner must delete and recreate the game. Acceptable for V1.

## Migration Plan

1. Add `JoinCode` (`VARCHAR(8) NOT NULL DEFAULT ''`) to the `Games` table via a new EF Core migration in `HexMaster.ThePrey.Games.Data`.
2. Back-fill existing games with a placeholder join code (existing games will be unreachable via the new join endpoint, which is acceptable — they predate the feature).
3. Deploy the API — old clients unaffected (no endpoint removed); new clients adopt the join flow.
4. Rollback: remove the migration, revert handler and DTO changes; no data loss since the column is additive.

## Open Questions

- Should the join endpoint return the full `GameDto` or just a lightweight "joined" confirmation? (Recommended: return the full DTO, consistent with other game state-changing operations.)
- Should the owner bypass the join code when joining their own game's lobby? (Recommended: yes — the owner is added automatically at creation or may join freely as lobby owner.)
