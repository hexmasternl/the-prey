## Why

After a player creates a game (the `maui-game-create-new` change navigates to the `game` route on `201 Created`), there is no lobby to land on — the `game` route still resolves to the placeholder `GamePage`. Players need a lobby where the owner can review and tweak the game settings, everyone can see who has joined and their role and readiness, the owner can pick the hunter, and other players can be invited via the game's secret pass code and a shareable deep link.

## What Changes

- Replace the placeholder `GamePage` (the existing `game` Shell route) with a real **game lobby page**. It resolves the current game itself (via the active-game endpoint → full-game read), so it works whether it was reached from create-success or the menu's Resume path.
- **Game code + share (top of page)**: show the game's secret pass code (`GameDto.GameCode`) verbatim, with a **Share** button next to it that invokes the device's **native share sheet** (WhatsApp, Messages, etc.). The shared payload is a short localized invite mentioning the pass code and a **deep link** that (when a receiver taps it) is intended to open the app on a join page.
- **Game settings**: show the five tuning selectors (Duration, Headstart Time, Duration Endgame, GPS Ping interval, GPS Ping at endgame) seeded from the game's current configuration. **Only the game owner** may change them; for non-owners they are read-only. Owner edits persist via authenticated `PUT /games/{id}/config`, reusing the create page's fixed option sets and the minutes→seconds conversion for the two ping intervals.
- **Participants list (bottom of page)**: one row per participant showing **name**, **role** (Hunter / Prey, derived from `HunterUserId`), and **ready state** (Ready / Not ready, from `IsReady`).
- **Designate the hunter**: tapping a participant makes that player the **hunter** and every other player a **prey**, via authenticated `POST /games/{id}/hunter` (in Lobby status this is the owner-only *designate-hunter* action). Only the owner may tap; for non-owners the rows are non-interactive.
- **Ready up (non-owners)**: every player **except the owner** gets a **SET READY** control that marks them ready via `POST /games/{id}/lobby/ready`; the owner never readies (the backend no-ops ready for the owner). Readiness is shown per participant in the list.
- **Settings change resets readiness**: whenever the owner changes a game parameter, **all non-owner ready flags reset to false** (enforced server-side by `UpdateSettings`) and every non-owner player must re-ready. The lobby reflects this from the returned/streamed snapshot.
- **Start the operation (owner)**: the owner sees a prominent **START OPERATION** button, enabled only when the game is startable — i.e. `GameDto.IsReadyToStart` is true (a hunter has been designated **and** every non-owner player is ready). Activating it sends authenticated `POST /games/{id}/start` with the designated hunter; on success the lobby hands off to the (separately-owned) gameplay screen via a navigation seam. Non-owners have no start button.
- **Live lobby updates**: the page reflects other players' joins, ready changes, hunter designation, settings edits, and game start in near-real-time by subscribing to the game's lobby event stream while visible (each event carries a full game snapshot); it unsubscribes when it disappears. An initial load populates the page immediately.
- Extend the game client seam (`IGameApiClient`) with the reads/writes the lobby needs — full-game read, update-settings, designate-hunter, set-ready, start-game — each mapping backend status codes to typed results, plus a lobby-stream seam and a native-share seam.

## Capabilities

### New Capabilities
- `maui-game-lobby`: The game lobby page reached from the `game` route — resolving the current game and loading its full state, showing the pass code, the owner-editable / non-owner-read-only settings selectors with the minutes→seconds ping conversion on save, the participants list (name / role / ready), the owner-only tap-to-designate-hunter action, the non-owner SET READY action, the reset of all non-owner readiness when the owner changes settings, the owner's START OPERATION action enabled only when `IsReadyToStart` is true, live updates while the page is visible, and the authenticated client calls (`GET /games/{id}`, `PUT /games/{id}/config`, `POST /games/{id}/hunter`, `POST /games/{id}/lobby/ready`, `POST /games/{id}/start`) with their result mapping.
- `maui-game-lobby-share`: The pass-code display and Share action — constructing the localized invite text and the join deep link from the game code, and handing it to the device's native share sheet, including the no-op when the user dismisses the share sheet.

## Impact

- **Depends on**:
  - `maui-game-create-new` — provides the `game`-route navigation target this page replaces, the `IGameApiClient` result-union client style, `IAccessTokenProvider`, and the five fixed option sets + minutes→seconds ping convention reused here for the settings selectors.
  - `main-menu-page` — provides the `game` Shell route and the "signed-in with an active game" Resume entry that also reaches this page.
  - Backend Games module contracts (already implemented, authoritative): `GET /games/{id}` (`GetGame`) → `GameDto` (`GameCode`, `Configuration`, `Participants[UserId, DisplayName, IsReady, State]`, `HunterUserId`, `OwnerUserId`, `IsOwnerPlayer`, `Status`, `IsReadyToStart`); `PUT /games/{id}/config` (`UpdateGameConfig`, owner-only, `403` for non-owner) with `UpdateGameSettingsRequest`; `POST /games/{id}/hunter` (`SetHunter`, in Lobby = owner designates hunter) with `SetHunterRequest(NewHunterUserId)`; `POST /games/{id}/lobby/ready` (`SetReady`, readies the caller, no-op for owner); `POST /games/{id}/start` (`StartGame`, owner-only, `Arm` enforces hunter-designated + all-non-owner-ready) with `StartGameRequest(HunterUserId)`; `GET /games/{id}/lobby/stream` (SSE, events carry a full `GameDto` snapshot); `GET /games/active` (`GetActiveGame`) to resolve the current game id. The domain's `UpdateSettings` resets all non-owner ready flags, and `IsReadyToStart` already encodes the start gate — the client reflects these, it does not re-implement them.
- **Client code** in `src/HexMaster.ThePrey.Maui.App`:
  - New `Pages/GameLobbyPage.xaml` (+ `.xaml.cs`) bound to the `game` route, replacing the placeholder `GamePage` as that route's destination; the settings selectors, participants list, pass-code + Share header, ready toggle, and busy/error regions.
  - New `ViewModels/GameLobbyViewModel.cs`: game resolution + full-state load, the five selectors seeded from config, owner-vs-non-owner editability, the ping minutes→seconds conversion on save, the participants projection (name / role / ready), the designate-hunter / set-ready / start commands with result mapping, `CanStart` derived from `IsReadyToStart`, live-snapshot handling, and the share command.
  - `Services/Api/IGameApiClient.cs` + `GameApiClient.cs`: add `GetGameAsync`, `UpdateGameSettingsAsync`, `DesignateHunterAsync`, `SetReadyAsync`, `StartGameAsync` and their typed results (`Success(GameDetails)` / `Validation` / `Forbidden` / `Unauthorized` / `NotFound` / `Error` as applicable), plus a `GameDetails` projection of `GameDto` (including `IsOwnerPlayer`, `HunterUserId`, `IsReadyToStart`, `Status`).
  - New `Services/Api/ILobbyStreamClient.cs` (+ impl): subscribe to `GET /games/{id}/lobby/stream`, yielding `GameDetails` snapshots; reconnect/teardown handled in the implementation.
  - New `Services/Platform/IShareService.cs` (+ impl) wrapping MAUI `Share.Default.RequestAsync(ShareTextRequest)`.
  - `Resources/Styles/*` + `Resources/Strings/*.resx`: styles for the header, selectors, participant rows, ready/role/hunter states, and localized strings for every label + the invite template — no inline visual literals, no hard-coded user-facing text.
  - `AppShell.xaml.cs` + `MauiProgram.cs`: point the `game` route at `GameLobbyPage`, register the page, view model, lobby-stream client, and share service.
- **Backend**: no changes. Reuses the existing Games-module endpoints.
- **Non-goals**: the countdown / in-progress **gameplay screens** and the onward-navigation destination after START OPERATION (separate change) — this page invokes the hand-off seam on start success but does not own what comes next; **inbound** deep-link handling and the **join page** the link targets (separate change) — this page only *constructs and shares* the link; removing/kicking a lobby player (`DELETE /games/{id}/lobby/{userId}`); the prey/hunter boundary-penalty toggles (left at their stored defaults, no UI); a map preview of the playfield; and free-entry durations/intervals beyond the fixed option sets.
